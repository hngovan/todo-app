# Todo App - Frontend

The client-side application for the Todo App, built with React 19, Vite, and Tailwind CSS v4.

## 🚀 Tech Stack

- **Framework**: React 19 + Vite
- **Styling**: Tailwind CSS v4
- **UI Components**: shadcn/ui, Radix UI
- **State Management**: Zustand
- **Forms & Validation**: react-hook-form, Zod
- **Networking**: Axios
- **Routing**: React Router DOM v7
- **i18n**: i18next, react-i18next

## ⚙️ Environment Setup

Ensure you have a `.env` file in the `frontend/` directory.

```dotenv
VITE_API_URL=http://localhost:3000
```

*(You can generate this from the root using `pnpm env:setup`)*

## � Quick Start

To run the frontend locally, first ensure you have the backend running.

```bash
cd frontend
pnpm install
pnpm dev
```

The application will be accessible at `http://localhost:5173`.

## �📜 Available Scripts

Run these commands from within the `frontend/` directory (or use `--filter` from the root):

```bash
# Starts the Vite development server
pnpm dev

# Compiles TypeScript and builds the app for production
pnpm build

# Runs ESLint to check for code issues
pnpm lint

# Runs Prettier to format source files
pnpm format

# Previews the production build locally
pnpm preview
```

## 📂 Project Structure

```text
src/
├── apis/           # API client integrations (Axios instances, endpoints)
├── components/     # Reusable React components
│   ├── common/     # Global shared components (Dialogs, Buttons, etc.)
│   └── ui/         # Base UI components generated by shadcn/ui
├── contexts/       # React Contexts (e.g., Theme, Auth)
├── hooks/          # Custom React hooks (e.g., useDebounce)
├── locales/        # i18n translation files (en, vi, etc.)
├── pages/          # Route-level components (Views)
├── types/          # Global TypeScript interfaces and types
├── utils/          # Helper functions and utilities
├── App.tsx         # Main Application wrapper
└── main.tsx        # React DOM entry point
```

## 🌐 API Integration

The frontend uses Axios to communicate with the NestJS backend. All external requests are heavily typed with internal generated schemas to provide end-to-end type safety.
